# 变更导航检查表

**目的:** 在BMad工作流中识别出重大变更(转向、技术问题、遗漏需求、失败的故事)时,系统性地引导选定的代理和用户完成所需的分析与规划.

**说明:** 与用户一起审查每一项.已完成/确认的标记为[x],不适用的标记为[N/A],或添加备注用于讨论点.

[[LLM: 初始化说明 - 变更导航

开发过程中的变更不可避免,但我们处理它们的方式决定了项目的成功或失败.

在继续之前,请理解:

1. 本检查表适用于影响项目方向的重大变更
2. 故事内的小调整不需要此流程
3. 目标是在适应新现实的同时尽量减少浪费的工作
4. 用户的支持至关重要 - 他们必须理解并批准变更

所需上下文:

- 触发的故事或问题
- 当前项目状态(已完成的故事、当前史诗)
- 访问PRD、架构和其他关键文档
- 了解剩余的计划工作

方法:
这是一个与用户的互动过程.一起通过每个部分,讨论影响和选项.用户做出最终决定,但提供有关技术可行性和影响的专业指导.

记住:变更不是失败,而是改进的机会.专业且建设性地处理它们.]]

---

## 1. 理解触发因素与背景

[[LLM: 首先充分理解发生了什么以及为什么发生.不要急于寻找解决方案.提出深入的问题:

- 到底发生了什么触发了这次审查？
- 这是一个一次性问题,还是更大问题的征兆？
- 这个问题是否能更早被预见？
- 哪些假设是错误的？

要具体且实事求是,而非指责导向.]]

- [ ] **识别触发故事:** 明确识别揭示问题的故事(或多个故事).
- [ ] **定义问题:** 精确阐述核心问题.
  - [ ] 是技术限制/死胡同吗？
  - [ ] 是新发现的需求吗？
  - [ ] 是对现有需求的根本误解吗？
  - [ ] 是基于反馈或新信息的必要转向吗？
  - [ ] 是失败/放弃的故事,需要新方法吗？
- [ ] **评估初步影响:** 描述直接观察到的后果(例如,阻碍进展、错误功能、不可行的技术).
- [ ] **收集证据:** 记录支持问题定义的任何特定日志、错误消息、用户反馈或分析.

## 2. 史诗影响评估

[[LLM: 变更会波及项目结构.系统性评估:

1. 我们能否通过修改保留当前史诗？
2. 鉴于此变更,未来的史诗仍有意义吗？
3. 我们是否在创建或消除依赖关系？
4. 史诗顺序需要重新排序吗？

考虑即时和下游影响.]]

- [ ] **分析当前史诗:**
  - [ ] 包含触发故事的当前史诗仍可完成吗？
  - [ ] 当前史诗是否需要修改(故事变更、添加、删除)？
  - [ ] 当前史诗是否应被放弃或根本重新定义？
- [ ] **分析未来史诗:**
  - [ ] 审查所有剩余的计划史诗.
  - [ ] 该问题是否需要更改未来史诗中计划的故事？
  - [ ] 该问题是否使任何未来史诗失效？
  - [ ] 该问题是否需要创建全新的史诗？
  - [ ] 未来史诗的顺序/优先级是否需要更改？
- [ ] **总结史诗影响:** 简要记录对项目史诗结构和流程的整体影响.

## 3. 工件冲突与影响分析

[[LLM: 文档在BMad中驱动开发.检查每个工件:

1. 此变更是否使已记录的决策无效？
2. 架构假设是否仍然有效？
3. 用户流程是否需要重新思考？
4. 技术约束是否与记录的不同？

要全面 - 遗漏的冲突会导致未来问题.]]

- [ ] **审查PRD:**
  - [ ] 该问题是否与PRD中陈述的核心目标或需求冲突？
  - [ ] PRD是否需要根据新理解进行澄清或更新？
- [ ] **审查架构文档:**
  - [ ] 该问题是否与记录的架构(组件、模式、技术选择)冲突？
  - [ ] 是否有特定的组件/图表/部分受到影响？
  - [ ] 技术列表是否需要更新？
  - [ ] 数据模型或模式是否需要修订？
  - [ ] 外部API集成是否受到影响？
- [ ] **审查前端规范(如适用):**
  - [ ] 该问题是否与FE架构、组件库选择或UI/UX设计冲突？
  - [ ] 是否有特定的FE组件或用户流程受到影响？
- [ ] **审查其他工件(如适用):**
  - [ ] 考虑对部署脚本、IaC、监控设置等的影响.
- [ ] **总结工件影响:** 列出所有需要更新的工件及所需变更的性质.

## 4. 前进路径评估

[[LLM: 清晰呈现选项及优缺点.对于每条路径:

1. 需要多少努力？
2. 会丢弃哪些工作？
3. 我们承担哪些风险？
4. 这对时间表有何影响？
5. 这从长远来看是否可持续？

诚实地看待权衡.几乎没有完美的解决方案.]]

- [ ] **选项1:直接调整/集成:**
  - [ ] 能否通过在现有计划内修改/添加未来故事来解决该问题？
  - [ ] 定义这些调整的范围和性质.
  - [ ] 评估此路径的可行性、努力和风险.
- [ ] **选项2:潜在回滚:**
  - [ ] 撤销已完成的故事是否会大大简化解决问题？
  - [ ] 指定考虑回滚的具体故事/提交.
  - [ ] 评估回滚所需的努力.
  - [ ] 评估回滚的影响(丢失的工作、数据影响).
  - [ ] 将净收益/成本与直接调整进行比较.
- [ ] **选项3:PRD MVP审查与潜在重新定范围:**
  - [ ] 鉴于问题和约束,原始PRD MVP是否仍可实现？
  - [ ] MVP范围是否需要缩减(移除功能/史诗)？
  - [ ] 核心MVP目标是否需要修改？
  - [ ] 是否需要替代方法来实现原始MVP意图？
  - [ ] **极端情况:** 该问题是否需要根本性重新规划,或可能需要新的PRD V2(由PM处理)？
- [ ] **选择推荐路径:** 根据评估,同意最可行的前进路径.

## 5. 冲刺变更提案组件

[[LLM: 提案必须可操作且清晰.确保:

1. 该问题以通俗语言解释
2. 影响尽可能量化
3. 推荐路径有清晰的依据
4. 下一步骤具体且已分配
5. 定义变更的成功标准

此提案指导所有后续工作.]]

(确保所有先前部分达成的一致点都包含在提案中)

- [ ] **识别的问题摘要:** 清晰简洁的问题陈述.
- [ ] **史诗影响摘要:** 史诗如何受到影响.
- [ ] **工件调整需求:** 需要更改的文档列表.
- [ ] **推荐的前进路径:** 所选解决方案及理由.
- [ ] **PRD MVP影响:** 对范围/目标的变更(如有).
- [ ] **高层次行动计划:** 故事/更新的下一步操作.
- [ ] **代理交接计划:** 识别所需角色(PM、架构师、设计架构师、PO).

## 6. 最终审查与交接

[[LLM: 变更需要协调.在结束前:

1. 用户是否完全认同计划？
2. 所有利益相关者是否理解影响？
3. 与其他代理的交接是否清晰？
4. 是否有回滚计划以防变更失败？
5. 我们将如何验证变更有效？

获得明确批准 - 隐含同意会导致问题.

最终报告:
完成检查表后,提供一份简洁的摘要:

- 改变了什么以及为什么
- 我们将如何处理
- 谁需要做什么
- 我们何时知道是否成功

保持行动导向和面向未来.]]

- [ ] **审查检查表:** 确认所有相关项目都已讨论.
- [ ] **审查冲刺变更提案:** 确保其准确反映讨论和决策.
- [ ] **用户批准:** 获得用户对提案的明确批准.
- [ ] **确认下一步骤:** 重申交接计划和具体代理的下一步行动.
