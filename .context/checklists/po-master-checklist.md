# 产品负责人(PO)主验证检查表

本检查表为产品负责人在开发执行前验证项目计划提供了一个全面的框架.它根据项目类型(绿场 vs 棕地)智能适应,并在适用时包含UI/UX考虑.

[[LLM: 初始化说明 - PO 主检查表

项目类型检测:
首先,通过检查确定项目类型:

1. 这是绿地项目(全新从头开始)吗？
   - 查看:新项目初始化,无现有代码库引用
   - 检查:prd.md、architecture.md、新项目设置故事

2. 这是棕地项目(增强现有系统)吗？
   - 查看:引用现有代码库,增强/修改语言
   - 检查:prd.md、architecture.md、现有系统分析

3. 项目是否包含UI/UX组件？
   - 检查:frontend-architecture.md、UI/UX规范、设计文件
   - 查看:前端故事、组件规范、用户界面提及

文档要求:
根据项目类型,确保您能访问以下资料:

对于绿场项目:

- prd.md - 产品需求文档
- architecture.md - 系统架构
- frontend-architecture.md - 如果涉及UI/UX
- 所有史诗和故事定义

对于棕地项目:

- prd.md - 棕地增强需求
- architecture.md - 增强架构
- 现有项目代码库访问(关键 - 无此无法继续)
- 当前部署配置和基础设施详细信息
- 数据库模式、API文档、监控设置

跳过说明:

- 对于绿场项目跳过标记为[[棕地仅限]]的部分
- 对于棕地项目跳过标记为[[绿场仅限]]的部分
- 对于仅后端项目跳过标记为[[UI/UX仅限]]的部分
- 在最终报告中注明所有跳过的部分

验证方法:

1. 深入分析 - 彻底分析每个项目与文档的匹配情况
2. 有据可依 - 在验证时引用具体章节或代码
3. 批判性思维 - 质疑假设并识别差距
4. 风险评估 - 考虑每个决策可能出现的问题

执行模式:
询问用户他们希望如何完成检查表:

- 逐部分进行(互动模式) - 审查每个部分,确认后再继续
- 一次性完成(综合模式) - 完成完整分析,最后呈现报告]]

[[LLM: 最终PO验证报告生成

   - 项目类型:[绿场/棕地] 带[UI/无UI]
   - 整体就绪度(百分比)
   - 通过/不通过建议
   - 关键阻塞性问题数量
   - 因项目类型而跳过的部分

   对于绿场:
   - 设置完整性
   - 依赖排序
   - MVP范围适当性
   - 开发时间表可行性

   对于棕地:
   - 集成风险等级(高/中/低)

   - 回滚准备情况

   - 按严重程度排列的前5大风险

   - 核心功能覆盖

   - 缺失的技术细节
   - [棕地] 集成点清晰度

6. 建议
   - 开发前必须修复的事项
   - 为质量应修复的事项
   - 可考虑改进的事项
   - MVP后推迟的事项

7. [棕地仅限] 集成信心

   - 集成点监控覆盖
   - 支持团队准备情况

- 任何失败部分的详细分析
