# 产品经理(PM)需求检查表

本检查表为一个全面的框架,确保产品需求文档(PRD)和史诗定义完整、结构良好且适合MVP开发.PM应在产品定义过程中系统性地完成每一项检查.

[[LLM: 初始化说明 - PM检查表

在继续本检查表之前,请确保您已获得以下资料:

1. prd.md - 产品需求文档(检查 .context/prd.md)
2. 任何用户研究、市场分析或竞争分析文档
3. 业务目标和策略文档
4. 任何现有的史诗定义或用户故事

重要:如果PRD缺失,请立即向用户询问其位置或内容,然后再继续.

验证方法:

1. 以用户为中心 - 每项需求都应追溯到用户价值
2. MVP聚焦 - 确保范围确实最小且可行
3. 清晰性 - 需求应明确且可测试
4. 完整性 - 覆盖产品愿景的所有方面
5. 可行性 - 需求在技术上可实现

执行模式:
询问用户他们希望如何完成检查表:

- 逐部分进行(互动模式) - 审查每个部分,提出发现,确认后再继续
- 一次性完成(综合模式) - 完成完整分析,最后呈现综合报告]]

## 1. 问题定义与背景

[[LLM: 任何产品的基础都是清晰的问题陈述.在审查本部分时:

1. 验证问题真实且值得解决
2. 检查目标受众是否具体,而非"所有人"
3. 确保成功指标可衡量,而非模糊的抱负
4. 寻找用户研究的证据,而非仅假设
5. 确认问题-解决方案匹配逻辑合理]]

### 1.1 问题陈述

- [ ] 清晰阐述要解决的问题
- [ ] 识别谁经历了该问题
- [ ] 解释为什么解决这个问题很重要
- [ ] 量化问题影响(如可能)
- [ ] 与现有解决方案的区分

### 1.2 业务目标与成功指标

- [ ] 定义了具体的、可衡量的业务目标
- [ ] 建立了明确的成功指标和KPI
- [ ] 指标与用户和业务价值挂钩
- [ ] 确定了基线测量(如适用)
- [ ] 指定了实现目标的时间框架

### 1.3 用户研究与洞察

- [ ] 明确定义了目标用户画像
- [ ] 记录了用户需求和痛点
- [ ] 总结了用户研究发现(如可用)
- [ ] 包含了竞争分析
- [ ] 提供了市场背景

## 2. MVP范围定义

[[LLM: MVP范围至关重要 - 范围太大会浪费资源,太小则无法验证.检查:

1. 这是否真正最小？挑战每个功能
2. 每个功能是否直接解决核心问题？
3. "可有可无"的功能是否明确与"必须"的功能分离？
4. 范围决策的依据是否已记录？
5. 您能在目标时间内交付吗？]]

### 2.1 核心功能

- [ ] 明确区分了必需功能与可有可无的功能
- [ ] 功能直接解决定义的问题陈述
- [ ] 每个史诗都与特定用户需求挂钩
- [ ] 功能和故事从用户角度描述
- [ ] 定义了成功所需的最低要求

### 2.2 范围边界

- [ ] 明确阐述了范围外的内容
- [ ] 包含了未来增强功能部分
- [ ] 记录了范围决策的理由
- [ ] MVP在最小化功能的同时最大化学习
- [ ] 范围已多次审查和改进

### 2.3 MVP验证方法

- [ ] 定义了测试MVP成功的方法
- [ ] 计划了初步用户反馈机制
- [ ] 指定了超越MVP的标准
- [ ] 阐明了MVP的学习目标
- [ ] 设定了时间期望

3. 可访问性不是事后考虑
4. 性能期望现实

### 3.2 易用性需求

### 3.3 UI需求

## 4. 功能需求

[[LLM: 功能需求必须足够清晰以便实施.检查:

1. 需求关注WHAT而非HOW(无实施细节)
2. 每项需求都是可测试的(QA如何验证？)
3. 依赖关系明确(哪些需要先构建？)
4. 需求使用一致的术语
5. 复杂功能被分解为可管理的部分]]

### 4.1 功能完整性

- [ ] 记录了MVP所需的所有功能
- [ ] 功能有清晰、以用户为中心的描述
- [ ] 指示了功能优先级/重要性
- [ ] 需求可测试和验证
- [ ] 识别了功能间的依赖关系

### 4.2 需求质量

- [ ] 需求具体且无歧义
- [ ] 需求关注WHAT而非HOW
- [ ] 需求使用一致的术语
- [ ] 复杂需求被分解为更简单的部分
- [ ] 最小化或解释了技术术语

### 4.3 用户故事与验收标准

- [ ] 故事遵循一致的格式
- [ ] 验收标准可测试
- [ ] 故事尺寸适当(不过大)
- [ ] 故事尽可能独立
- [ ] 故事包含必要的上下文
- [ ] 为相关后端/数据故事在ACs中定义了本地可测试性要求(例如通过CLI)

## PRD与史诗验证摘要

[[LLM: 最终PM检查表报告生成

创建一份综合验证报告,包括:

1. 执行摘要
   - PRD完整性总体百分比
   - MVP范围适当性(太大/刚好/太小)
   - 进入架构阶段的准备情况(就绪/接近就绪/未就绪)
   - 最关键的缺口或关切

2. 类别分析表
   填写实际表格:
   - 状态:通过(90%+完成)、部分通过(60-89%)、失败(<60%)
   - 关键问题:阻碍进展的具体问题

3. 优先级最高的问题
   - 阻塞项:架构师开始工作前必须修复
   - 高:为质量应修复
   - 中:将提高清晰度
   - 低:很好有

4. MVP范围评估
   - 可能为真正MVP而削减的功能
   - 缺失的关键功能
   - 复杂性问题
   - 时间线现实性

5. 技术就绪度
   - 技术约束的清晰度
   - 识别的技术风险
   - 需要架构师调查的领域

6. 建议
   - 解决每个阻塞项的具体行动
   - 建议的改进
   - 下一步

在呈现报告后,询问用户是否希望:

- 任何失败部分的详细分析
- 改进特定领域的建议
- 帮助完善MVP范围]]
