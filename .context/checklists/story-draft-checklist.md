# 故事草稿检查表

Scrum Master 应使用此检查表验证每个故事是否包含足够的上下文,以便开发代理能够成功实施,同时假定开发代理具有合理的推理能力.

[[LLM: 初始化说明 - 故事草稿验证

在继续本检查表之前,请确保您能访问以下资料:

1. 正在验证的故事文档(通常在 .context/stories/ 中或直接提供)
2. 上级PRD上下文
3. 任何引用的架构或设计文档
4. 如果此故事建立在先前工作基础上,则包括之前的关联故事

重要:本检查表在实施开始前验证单个故事.

验证原则:

1. 清晰性 - 开发者应清楚要构建什么
2. 上下文 - 为什么构建以及如何融入整体
3. 指导 - 关键技术决策和应遵循的模式
4. 可测试性 - 如何验证实现是否有效
5. 自包含 - 大部分所需信息都在故事本身中

记住:我们假设有能力的开发代理能:

- 研究文档和代码库
- 做出合理的技术决策
- 遵循既定模式
- 在真正卡壳时寻求澄清

我们检查的是充分的指导,而不是详尽的细节.]]

## 1. 目标与上下文清晰度

[[LLM: 没有明确的目标,开发者就会构建错误的东西.验证:

1. 故事清楚地说明了要实现的功能
2. 商业价值或用户收益清晰
3. 说明了它如何融入更大的PRD/产品
4. 依赖关系明确("需要故事X完成")
5. 成功标准具体,而非模糊]]

- [ ] 故事目标/目的已清晰说明
- [ ] 与PRD目标的关系显而易见
- [ ] 说明了故事如何融入整体系统流程
- [ ] 识别了与先前故事的依赖关系(如适用)
- [ ] 商业背景和价值明确

## 2. 技术实施指导

[[LLM: 开发者需要足够的技术上下文来开始编码.检查:

1. 提及了需要创建或修改的关键文件/组件
2. 在非显而易见时指定了技术选择
3. 识别了与现有代码的集成点
4. 定义或引用了数据模型或API契约
5. 指出了非标准模式或例外情况

注意:我们不需要列出每个文件 - 只列出重要的即可.]]

- [ ] 已识别需要创建/修改的关键文件(不一定是详尽的)
- [ ] 提及了此故事所需的具体技术
- [ ] 关键API或接口有足够描述
- [ ] 引用了必要的数据模型或结构
- [ ] 列出了所需环境变量(如适用)
- [ ] 注明了任何对标准编码模式的例外情况

## 3. 参考有效性

[[LLM: 参考应有帮助,而不是变成寻宝游戏.确保:

1. 参考指向特定部分,而非整个文档
2. 解释了每个参考的相关性
3. 关键信息在故事中总结
4. 参考可访问(无断裂链接)
5. 如果需要,总结了之前的故事上下文]]

- [ ] 对外部文档的引用指向特定相关部分
- [ ] 之前故事的关键信息已总结(而非仅引用)
- [ ] 提供了参考相关性的上下文
- [ ] 参考使用一致格式(例如,`docs/filename.md#section`)

## 4. 自包含评估

[[LLM: 故事应基本自包含,以避免上下文切换.验证:

1. 核心要求在故事中,而非仅在参考中
2. 领域术语已解释或从上下文中显而易见
3. 假设已明确说明
4. 提及了边缘情况(即使推迟)
5. 故事可在不阅读10个其他文档的情况下理解]]

- [ ] 所需的核心信息已包含(不过度依赖外部文档)
- [ ] 隐含假设已明确说明
- [ ] 领域特定术语或概念已解释
- [ ] 已处理边缘情况或错误场景

## 5. 测试指导

[[LLM: 测试确保实现确实有效.检查:

1. 指定了测试方法(单元、集成、端到端)
2. 列出了关键测试场景
3. 成功标准可衡量
4. 注明了特殊测试考虑
5. 故事中的验收标准可测试]]

- [ ] 概述了所需测试方法
- [ ] 识别了关键测试场景
- [ ] 定义了成功标准
- [ ] 注明了特殊测试考虑(如适用)

## 验证结果

[[LLM: 最终故事验证报告

生成一份简洁的验证报告:

1. 快速摘要
   - 故事就绪度:就绪/需要修订/阻塞
   - 清晰度评分(1-10)
   - 识别的主要差距

2. 填写验证表格:
   - 通过:要求明确满足
   - 部分通过:存在一些差距但可行
   - 失败:关键信息缺失

3. 具体问题(如有)
   - 列出需要修复的具体问题
   - 建议具体改进
   - 识别任何阻塞性依赖

4. 开发者视角
   - 作为开发者,您能否按此实施此故事？
   - 您会有哪些问题？
   - 什么可能导致延迟或返工？

务实对待 - 完美文档不存在,但它必须足以提供开发代理完成工作所需的极端上下文,而不会造成混乱.]]
